## 意思決定ログ
| 選択肢 | 採用 | 理由 |
|--------|------|------|
| TRANSCRIPT_RECEIVED を VoiceAction に追加 vs 別 state で管理 | VoiceAction に追加 | 既存の reducer パターンに統一。別 state だと hook の戻り値管理が複雑化 |
| DebugTranscriptOverlay を React.memo 化 | 採用 | TRANSCRIPT_RECEIVED が interim result のたびに dispatch されるため、memo 化で不要な再レンダリングを防止 |
| ListeningOverlay の transcript 表示位置 | カウントダウン下 | Ready テキスト + カウントダウンの文脈の延長で自然な配置 |
| IDLE 遷移時の lastTranscript クリア | 採用 | IDLE に戻った時点で前回の認識結果は不要。次のウェイクワード待機で新しい結果が来る |

## 学び（Lessons Learned）
- WSL2 環境での Jest 実行は TMPDIR の設定と sandbox 制限に注意が必要。`dangerouslyDisableSandbox: true` + `--forceExit` の組み合わせが安定
- TRANSCRIPT_RECEIVED は状態遷移を伴わないメタデータ更新アクションという新パターン。reducer の switch 文の default の直前に配置することで、既存の状態遷移ロジックへの影響を最小化できた
- 予測と実際の乖離がほぼゼロ。Plan モードでの十分なコード調査が予測精度を高めた
