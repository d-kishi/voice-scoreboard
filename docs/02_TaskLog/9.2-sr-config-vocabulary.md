## 意思決定ログ
| 選択肢 | 採用 | 理由 |
|--------|------|------|
| isWakeword に「ヘイスコア」判定を明示追加 | 不採用 | 既存の `includes('スコア')` で「ヘイスコア」がカバーされ、`includes('すこあ')` で「へいすこあ」もカバー済み。冗長なコード追加は不要 |
| COMMAND_MAP で長い keyword を先に配置 | 採用 | 「右チーム」を「右」より前に配置。includes マッチなので実動作への影響はないが、意図を明確にするコード構造上の判断 |
| volumechange の RMS dB を UI に表示 | 不採用（ログのみ） | Task 9.1 の TRANSCRIPT_RECEIVED パターンに追従して UI 表示する方法もあるが、RMS dB は logcat で十分。過剰な state 更新を避ける |
| `requiresOnDeviceRecognition: true` を全モードに適用 | 採用 | wakeword/command 両方でオンデバイス認識を明示。Android 12 以下では無視されるだけでリスクなし |

## 学び（Lessons Learned）
- 既存の includes マッチングが想定以上に多くのバリアントをカバーしていた。「ヘイスコア」は `includes('スコア')` で自動的にマッチするため、isWakeword への明示追加は不要だった。新しいウェイクワードを検討する際は、既存マッチパターンとの重複を先に確認すべき
- contextualStrings の拡張と COMMAND_MAP の拡張は独立した変更であり、テストも別 describe ブロックで整理できた。TDD の RED フェーズで 11 テスト中 9 テストが既にパスし、実際に失敗したのは「取り消し」「とりけし」の 2 テストのみだった
