# Task 8.5: 音声認識ノイズ耐性改善

## 実装予測（Plan モードでの分析）
- バレーボール練習での実地検証で、周囲の会話音が混入しウェイクワード/コマンドが検知不能と判明
- 4層の改善を計画: Layer 1（interim result）、Layer 2（subsequence matching）、Layer 3（長文エイリアス）、Layer 4（ウェイクワード長文化）
- Layer 1/2 は即時実装、Layer 3/4 は GitHub Issues として起票
- Agent Teams で3エージェント並列実装（layer1-impl / layer2-impl / issues-creator）
- 変更ファイルが独立しているためコンフリクトなし

## 実施内容
- tasks.md の Task 8.5: 音声認識ノイズ耐性改善
- Layer 1: ウェイクワード検知の interim result 対応
- Layer 2: 部分列マッチング + 距離制約によるノイズ耐性強化
- Layer 3/4: GitHub Issues 起票（将来タスク）

## 意思決定ログ
| 選択肢 | 採用 | 理由 |
|--------|------|------|
| interim result での判定（Layer 1） | 採用 | 誤検知リスク低（LISTENING 10秒タイムアウトに入るだけ）。騒音環境では isFinal 待ちで遅延が大きくなる |
| maxGap=2（Layer 2） | 採用 | 「各文字間に最大2文字の雑音」を許容。3以上だと誤検知リスクが上がる |
| ギャップ超過時のリセット再探索 | 採用 | テキスト中盤でターゲットが現れるケースに対応。単純打ち切りだと見逃す |
| Layer 3/4 を Issues に分離 | 採用 | Layer 1/2 の効果を実地検証してから判断。過剰実装を避ける |
| Agent Teams 3並列 | 採用 | 変更ファイルが独立しており、コンフリクトリスクゼロ。実装時間を大幅短縮 |

## 学び（Lessons Learned）
- バレーボール練習の騒音環境では、`includes()` ベースの連続部分文字列マッチングでは不十分。雑音が文字間に混入するため、部分列（subsequence）マッチングが有効
- interim result（isFinal=false）でのウェイクワード検知は、騒音環境で特に有効。雑音が混入する前のクリーンな部分結果で早期検知できる
- Agent Teams の並列実装は、変更ファイルが独立している場合に非常に効果的。3タスクを同時に実行し、全体の実装時間を短縮できた
- `isSubsequenceMatch` のギャップ超過時リセット再探索は、テキスト中盤にターゲットが現れるエッジケースで重要
